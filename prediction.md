I expect that for smaller lists insertion sort should be faster, since it has low overhead, and does not use additional arrays or recursion. Technically, merge sort does have a better time complexity of O(nlog2(n)) but it has extra overhead because it involves recursion, and merging sublists and using extra memory and splitting the list. So if you have a very small list, for example with 5 elements, insertion sort just compares the nearby elements and shifts them if it's needed, but merge sort on the other hand splits the list, calls the function recursively, makes smaller lists, merges them together again, and then finally gets the sorted list.

I think that for random data maybe whilst n is small, maybe somewhere around n= 100 would be where the crossover happens. For data that is sorted already, I think n will probably be a bit higher since it is already sorted and merge sort just adds overhead, whilst insertion sort would just go over the list and check if all the items are in order and it doesn't have to shift anything so it's more like O(n) rather than O(n^2). However, for reversely sorted data, I would imagine that merge sort starts winning in terms of speed at around perhaps less than 50 but greater than 20. The reason I think this is because it has to go through many elements, so the worst case scenario of elements. So compared to random data and sorted data, merge sort will win relatively faster. For data with lots of duplicates, perhaps insertion sort will perform better than on random data, but not as good as already sorted data and the reason I think this is because lots of duplicates make it so that insertion sort doesn't have to shift as much.
